# Assumptions

- **Environment setup:** The application is assumed to run locally or on a staging URL with the React frontend served separately from (or proxied to) the Spring Boot backend, with both reachable from the test machine (e.g., frontend at `http://localhost:3000`, backend API at `http://localhost:8080/api`). Selectors and endpoint paths in the automation scripts will need to be updated to match the actual running application.
- **JWT behavior:** It is assumed the JWT is returned in the login response body (or as an HTTP-only cookie) and is required as a Bearer token (or via cookie) on all protected endpoints, including the Dashboard data-fetching API. The exact storage mechanism (localStorage vs. cookie) affects how automation scripts should verify session state.
- **Token expiration:** It is assumed JWTs have a defined expiration time (commonly 15 minutes to a few hours) after which the token becomes invalid and the user is forced to log in again; exact expiry duration was not specified in the assignment and should be confirmed against the actual backend configuration.
- **Validation rules:** Field validation (name length, password complexity, email format) is assumed to be enforced both on the frontend (for immediate user feedback) and the backend (as the source of truth), per the "Validation Notes" in the assignment. Test cases assume backend validation is authoritative even if UI validation is bypassed.
- **Rate limiting:** The 10 attempts/hour/IP limit is assumed to apply per source IP address and to reset on a rolling or fixed 60-minute window; the assignment does not specify which, so this should be confirmed with the actual implementation before finalizing BC-16.
